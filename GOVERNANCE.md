# 太极 API 社区治理模式

> **核心理念**: 社区共同所有、共同决策、共同维护、共同受益
>
> **创始人保留条款**: 项目创始人保留最高管理权限和最终决策权

---

## 👑 创始人权限

### 创始人定义

**创始人**: [@ad7007](https://github.com/ad7007) - 余慕名

项目发起人和初始开发者，拥有特殊地位和权限。

### 创始人权限

#### 最高管理权
- ✅ **项目方向决策权** - 决定项目整体发展方向
- ✅ **最终否决权** - 可否决任何社区决策
- ✅ **紧急处置权** - 紧急情况下的快速决策权
- ✅ **维护者任命/罢免权** - 直接任命或罢免维护者
- ✅ **许可证变更权** - 决定项目许可证
- ✅ **品牌使用权** - 项目名称和商标使用

#### 特殊权限实现
- ✅ GitHub 仓库 Owner 权限（不可转让）
- ✅ PyPI 包发布权限
- ✅ 域名和网站控制权
- ✅ 官方社交媒体控制权

### 创始人权限限制

为保障社区利益，创始人权限受以下约束：

- ⚠️ **重大决策需公示** - 变更许可证等重大决定需提前 30 天公示
- ⚠️ **社区监督** - 创始人决策接受社区监督
- ⚠️ **透明度要求** - 财务相关决策需公开
- ⚠️ **继任计划** - 如创始人无法继续，需指定继任者

### 创始人 vs 社区

| 决策类型 | 决策方式 | 创始人权限 |
|---------|---------|-----------|
| 日常 PR 合并 | 维护者决定 | 不干预 |
| 功能新增 | 社区投票（2/3） | 可否决 |
| 架构变更 | 社区投票（2/3） | 可否决 |
| 许可证变更 | 创始人决定 + 公示 | 最终决定 |
| 维护者任命 | 社区投票 + 创始人确认 | 最终任命 |
| 项目方向 | 社区讨论 + 创始人决定 | 最终决定 |

## 🌟 治理原则

### 1. 去中心化 + 创始人监督

- ✅ **社区共同所有** - 属于所有贡献者
- ✅ **开放决策** - 重大决定由社区投票
- ⚠️ **创始人保留最终决策权** - 保障项目方向

### 2. 透明公开

- ✅ 所有讨论在 GitHub 公开
- ✅ 所有决策有记录
- ✅ 所有代码可审查
- ✅ 创始人决策需说明理由

### 3. 任人唯贤

- ✅ 贡献决定影响力
- ✅ 能力决定责任
- ✅ 公平对待每个贡献者
- ⚠️ 创始人有权破格提拔

### 4. 可持续发展

- ✅ 避免个人负担过重
- ✅ 培养新维护者
- ✅ 长期规划
- ✅ 创始人继任计划

## 🗳️ 决策机制

### 日常决策

**范围**: 
- Bug 修复
- 文档更新
- 小功能改进

**流程**:
```
提交 PR → 维护者 Review → 合并
```

**通过条件**: 1 个维护者同意

**创始人权限**: 不干预（除非发现问题）

---

### 中等决策

**范围**:
- 新功能
- API 变更
- 性能优化

**流程**:
```
提交 Issue 讨论 → 提交 PR → 2 个维护者 Review → 合并
```

**通过条件**: 2 个维护者同意

**创始人权限**: 可否决（需说明理由）

---

### 重大决策

**范围**:
- 架构变更
- 许可证修改
- 治理模式变更
- 核心维护者任命

**流程**:
```
提交 RFC → 社区讨论 7 天 → 投票 7 天 → 创始人确认 → 执行
```

**通过条件**: 
- 投票率 > 50%
- 赞成票 > 2/3
- **创始人最终确认**

**创始人权限**:
- ✅ 可否决（需书面说明理由）
- ✅ 可要求重新投票
- ✅ 可提出替代方案

## 📋 投票规则

### 投票资格

- ✅ 活跃贡献者（5+ PR）
- ✅ 维护者
- ✅ 核心维护者

**权重**: 1 人 1 票（平等）

### 投票流程

1. **提案**: 在 Issue 中提交 RFC
2. **讨论**: 7 天公开讨论
3. **投票**: 7 天投票期
4. **结果**: 公布并执行

### 投票类型

| 类型 | 通过条件 | 适用范围 |
|------|---------|---------|
| 简单多数 | > 50% 赞成 | 日常决策 |
| 绝对多数 | > 2/3 赞成 | 重大决策 |
| 一致同意 | 100% 赞成 | 许可证变更 |

## 🔄 版本发布流程

### 版本号规则

遵循语义化版本：`MAJOR.MINOR.PATCH`

- **MAJOR**: 不兼容的 API 变更
- **MINOR**: 向后兼容的功能新增
- **PATCH**: 向后兼容的问题修复

### 发布流程

```
1. 创建 Release Branch
   ↓
2. 更新版本号
   ↓
3. 更新 CHANGELOG
   ↓
4. 维护者 Review (2 人同意)
   ↓
5. 打 Tag
   ↓
6. 发布到 PyPI
   ↓
7. 创建 GitHub Release
   ↓
8. 社区公告
```

### 发布频率

- **PATCH**: 随时（Bug 修复）
- **MINOR**: 每月 1 次
- **MAJOR**: 每季度 1 次
