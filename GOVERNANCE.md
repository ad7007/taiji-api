# 太极 API 社区治理模式

> **核心理念**: 社区共同所有、共同决策、共同维护、共同受益

---

## 🌟 治理原则

### 1. 去中心化

- ❌ **没有唯一维护者** - 不依赖任何个人
- ✅ **社区共同所有** - 属于所有贡献者
- ✅ **开放决策** - 重大决定由社区投票

### 2. 透明公开

- ✅ 所有讨论在 GitHub 公开
- ✅ 所有决策有记录
- ✅ 所有代码可审查

### 3. 任人唯贤

- ✅ 贡献决定影响力
- ✅ 能力决定责任
- ✅ 公平对待每个贡献者

### 4. 可持续发展

- ✅ 避免个人负担过重
- ✅ 培养新维护者
- ✅ 长期规划

---

## 👥 角色与职责

### 用户 (User)

**定义**: 使用太极 API 的人

**权益**:
- ✅ 免费使用所有功能
- ✅ 报告 Bug 和提出建议
- ✅ 参与社区讨论

**责任**:
- ⚪ 遵守使用规范
- ⚪ 尊重其他成员

---

### 贡献者 (Contributor)

**定义**: 提交过至少 1 个 PR 并被合并的人

**权益**:
- ✅ 名字列入 CONTRIBUTORS.md
- ✅ 社区认可徽章
- ✅ 优先获得帮助

**责任**:
- ✅ 保证代码质量
- ✅ 参与 Code Review

**成为方式**:
1. 提交 PR
2. 通过审核
3. 自动成为

---

### 活跃贡献者 (Active Contributor)

**定义**: 提交过至少 5 个 PR 的贡献者

**权益**:
- ✅ 贡献者所有权益
- ✅ 优先 Review 权
- ✅ 参与月度会议
- ✅ 维护者提名资格

**责任**:
- ✅ 每季度至少 1 个贡献
- ✅ 帮助新贡献者

**成为方式**:
- 累计提交 5 个 PR
- 自动升级

---

### 维护者 (Maintainer)

**定义**: 被社区认可的代码合并权限持有者

**权益**:
- ✅ 代码合并权
- ✅ Issue 管理权
- ✅ 项目决策投票权
- ✅ 官方认可证书
- ✅ 社区影响力

**责任**:
- ✅ 每周至少投入 2 小时
- ✅ Review 和合并 PR
- ✅ 回答社区问题
- ✅ 参与项目规划

**成为方式**:
1. 提交申请（在 Issue 中）
2. 展示贡献历史（至少 10 个 PR）
3. 现有维护者投票（2/3 同意）
4. 正式任命

**卸任方式**:
- 主动辞职
- 连续 3 个月无贡献
- 社区弹劾（1/2 维护者同意）

---

### 核心维护者 (Core Maintainer)

**定义**: 负责项目整体方向的资深维护者

**权益**:
- ✅ 维护者所有权益
- ✅ 最终决策权（争议时）
- ✅ 代表项目对外交流

**责任**:
- ✅ 每周至少投入 5 小时
- ✅ 协调维护者工作
- ✅ 制定项目规划
- ✅ 处理社区争议

**成为方式**:
- 维护者投票（3/4 同意）
- 社区公示 7 天无异议

---

## 🗳️ 决策机制

### 日常决策

**范围**: 
- Bug 修复
- 文档更新
- 小功能改进

**流程**:
```
提交 PR → 维护者 Review → 合并
```

**通过条件**: 1 个维护者同意

---

### 中等决策

**范围**:
- 新功能
- API 变更
- 性能优化

**流程**:
```
提交 Issue 讨论 → 提交 PR → 2 个维护者 Review → 合并
```

**通过条件**: 2 个维护者同意

---

### 重大决策

**范围**:
- 架构变更
- 许可证修改
- 治理模式变更
- 核心维护者任命

**流程**:
```
提交 RFC → 社区讨论 7 天 → 投票 7 天 → 执行
```

**通过条件**: 
- 投票率 > 50%
- 赞成票 > 2/3

---

## 📋 投票规则

### 投票资格

- ✅ 活跃贡献者（5+ PR）
- ✅ 维护者
- ✅ 核心维护者

**权重**: 1 人 1 票（平等）

### 投票流程

1. **提案**: 在 Issue 中提交 RFC
2. **讨论**: 7 天公开讨论
3. **投票**: 7 天投票期
4. **结果**: 公布并执行

### 投票类型

| 类型 | 通过条件 | 适用范围 |
|------|---------|---------|
| 简单多数 | > 50% 赞成 | 日常决策 |
| 绝对多数 | > 2/3 赞成 | 重大决策 |
| 一致同意 | 100% 赞成 | 许可证变更 |

---

## 🔄 版本发布流程

### 版本号规则

遵循语义化版本：`MAJOR.MINOR.PATCH`

- **MAJOR**: 不兼容的 API 变更
- **MINOR**: 向后兼容的功能新增
- **PATCH**: 向后兼容的问题修复

### 发布流程

```
1. 创建 Release Branch
   ↓
2. 更新版本号
   ↓
3. 更新 CHANGELOG
   ↓
4. 维护者 Review (2 人同意)
   ↓
5. 打 Tag
   ↓
6. 发布到 PyPI
   ↓
7. 创建 GitHub Release
   ↓
8. 社区公告
```

### 发布频率

- **PATCH**: 随时（Bug 修复）
- **MINOR**: 每月 1 次
- **MAJOR**: 每季度 1 次

## 🛡️ 冲突解决

### 争议处理流程

```
1. 当事方协商
   ↓ (失败)
2. 维护者调解
   ↓ (失败)
3. 核心维护者仲裁
   ↓ (失败)
4. 社区投票
```

### 行为准则

所有社区成员必须遵守 [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md)

**违规处理**:
- 首次：警告
- 再次：暂停参与权（30 天）
- 三次：永久禁止
